# 画面設計の設計判断

---

## 1. UI コンポーネントライブラリ

[Question] UI コンポーネントライブラリは何を使いますか？

- **A: shadcn/ui**: Radix UI ベース。コンポーネントをプロジェクトにコピーして使うスタイル。カスタマイズ性が高く、Tailwind CSS との相性が良い。トレンドとして人気
- **B: MUI (Material UI)**: Google Material Design ベース。業務アプリでの実績が豊富。コンポーネントが充実
- **C: Ant Design**: エンタープライズ向け。テーブル・フォームが強力。中国系プロジェクトで多い
- **D: Tailwind CSS のみ**: ライブラリなし。全て自前で組む。デモとしては工数が大きい

提案: **A（shadcn/ui）**。Tailwind CSS と合わせて使える。コンポーネントがプロジェクト内にあるので中身が見えて学習にも良い。

[Answer]

A

---

## 2. レイアウト構成

[Question] 画面のレイアウトはどうしますか？

- **A: サイドバー + メインコンテンツ**: 左にナビゲーション、右にコンテンツ。管理画面の定番
- **B: トップナビ + メインコンテンツ**: 上部にナビゲーションバー、下にコンテンツ。画面幅を広く使える
- **C: ロール別レイアウト**: 一般社員はシンプル（トップナビ）、管理者は管理画面（サイドバー）

提案: **A（サイドバー型）**。業務アプリとして自然で、ロールに応じてメニュー項目を出し分ければ OK。

[Answer]

A

---

## 3. 状態管理

[Question] サーバーステートの管理に使うライブラリはどちらですか？

- **A: SWR**: Vercel 製。Next.js との親和性が高い。軽量でシンプル
- **B: TanStack Query (React Query)**: 機能が豊富（楽観的更新、無限スクロール等）。コミュニティが大きい

提案: **A（SWR）**。Next.js プロジェクトでは自然な選択。デモ規模では SWR の機能で十分。

[Answer]

B

---

## 4. フォームバリデーション

[Question] フォームのバリデーションライブラリはどうしますか？

- **A: React Hook Form + Zod**: 型安全なバリデーション。パフォーマンスが良い。デファクトスタンダード
- **B: Formik + Yup**: 古くからある定番。やや冗長だが情報が多い

提案: **A（React Hook Form + Zod）**。TypeScript との相性が良く、モダンな選択。

[Answer]

A
