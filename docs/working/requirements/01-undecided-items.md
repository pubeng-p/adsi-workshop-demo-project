# 未決事項の解消

project-brief.md の未決事項を Q&A 形式で決定していく。

---

## 1. 認証方式

[Question] 社員の認証方式はセッションベースと JWT のどちらにしますか？

- **セッションベース**: Spring Security のデフォルト。サーバー側でセッション管理。Spring Boot 4.x 移行も自然。実装がシンプル
- **JWT**: ステートレス。スケーラビリティに優れるが、トークン失効の管理が複雑。デモでは冗長な構成

[Answer]

セッションベース

---

## 2. 社員マスタの管理方法

[Question] 社員マスタの管理方法はどうしますか？

- **管理画面で登録**: 管理者が画面から社員を追加・編集・削除する。CRUD のデモにもなる
- **初期データ投入のみ**: Flyway のマイグレーションや seed スクリプトで投入。管理画面は作らない
- **両方**: 初期データ + 管理画面

[Answer]

両方用意する

---

## 3. 打刻の方法

[Question] 出勤・退勤の打刻方法はどうしますか？

- **ボタン押下のみ**: 画面上の「出勤」「退勤」ボタンを押すだけ。シンプル
- **ボタン押下 + 位置情報**: 打刻時に GPS 座標を記録する。不正打刻の防止に使える
- **ボタン押下 + コメント入力**: 打刻時に任意のコメント（遅刻理由等）を入力できる

[Answer]

ボタン押下

---

## 4. 勤怠修正の承認フロー

[Question] 打刻忘れや誤りの修正はどういうフローにしますか？

- **管理者が直接修正**: 管理者が該当社員の勤怠を直接編集する。承認フローなし
- **社員が申請 → 管理者が承認**: 社員が修正リクエストを出し、管理者が承認/却下する
- **社員が申請 → 上長が承認**: 組織階層を使った承認フロー（部署構造が必要）

[Answer]

社員が申請→上長が承認

---

## 5. 勤務時間の計算ルール

[Question] 勤務時間の計算ルールはどうしますか？

### 5-1. 休憩時間の扱い

- **固定控除**: 例えば 6 時間以上勤務で 45 分、8 時間以上で 60 分を自動控除
- **打刻制**: 休憩開始・終了も打刻する
- **なし**: 休憩時間は考慮しない（出勤〜退勤の差分がそのまま勤務時間）

[Answer]

固定控除

### 5-2. 残業の定義

- **固定閾値**: 例えば 1 日 8 時間を超えた分が残業
- **なし**: 残業の区分は設けない（勤務時間の記録のみ）

[Answer]

固定閾値

---

## 6. 部署・組織の階層構造

[Question] 部署や組織の階層構造は必要ですか？

- **フラット**: 部署は 1 階層のみ（例: 「開発部」「営業部」）。社員は 1 つの部署に所属
- **階層あり**: 部署の親子関係がある（例: 「本部 > 部 > 課」）
- **なし**: 部署の概念を持たない。管理者は全社員を管理する

[Answer]

フラット

---

## 7. 月次集計の出力形式

[Question] 月次集計の出力形式はどうしますか？

- **画面表示のみ**: 管理画面に一覧表示する
- **画面表示 + CSV エクスポート**: 一覧表示に加えて CSV ダウンロード機能を提供
- **画面表示 + CSV + PDF**: 帳票としての出力も行う

[Answer]

画面表示 + CSV エクスポート + PDF
SIer系のプロジェクトでよく使われる帳票出力ライブラリも試したい。

### 補足 A: 承認フローと組織構造の整合性

Q4 で「社員が申請→上長が承認」、Q6 で「フラット（1 階層）」と回答。

[Question] フラット部署での「上長」は「その部署の管理者ロール」= 上長、という理解で合っていますか？（例: 開発部に所属する社員の修正申請は、開発部の管理者が承認する）

[Answer]
管理者ロールについて何ができるか詳しく設定するべきかもしれない。
上長は、ある部署に課長や部長みたいな概念があり、その人が承認するイメージだった。
社員登録は上長じゃなくてこのアプリの管理者（現実に照らし合わせると人事部門の担当者）が行うイメージだった。

### 補足 D: 帳票出力ライブラリ

Q7 で「SIer 系でよく使われる帳票出力ライブラリも試したい」とのこと。

[Question] 帳票出力ライブラリに好みはありますか？

- **JasperReports**: SIer 定番。テンプレートベースで複雑な帳票も対応可能。学習コストはやや高い
- **Apache PDFBox**: 軽量。プログラムで直接 PDF を組み立てる。シンプルな帳票向き
- **OpenPDF (iText のフォーク)**: iText 2 系の OSS フォーク。比較的シンプルな API
- **おまかせ**: デモに適したものを AI に選んでほしい

[Answer]
JasperReports を試してみたいです。

### 補足 F: ロール体系の整理

補足 A・E の回答から、ロールが当初の 2 種類（一般社員 / 管理者）から 3 種類になりそう。

[Question] 以下の整理で合っていますか？

| ロール | 誰？ | できること |
| --- | --- | --- |
| 一般社員 | 一般の従業員 | 自分の出退勤打刻、自分の勤怠履歴確認、勤怠修正の申請 |
| 上長 | 部署の課長・部長相当 | 一般社員の権限 + 自部署メンバーの勤怠確認 + 修正申請の承認/却下 |
| 管理者 | 人事部門の担当者 | 社員マスタの CRUD + 全社員の勤怠確認 + 月次集計・帳票出力 |

補足:

- 「上長」はロール項目ではなく、部署ごとの「上長フラグ」で管理する想定（1 部署に 1 人の上長）
- 管理者は特定の部署に所属しない（or 人事部に所属する）システム横断のロール
- 上長は自部署のみ管理。他部署は見えない

認識が違うところがあれば修正してください。

[Answer]

あっています。

### 補足 G: 上長が不在の場合

[Question] 上長が休暇や退職で不在の場合、修正申請の承認はどうなりますか？

- **管理者が代行**: 管理者（人事）が代わりに承認する
- **考慮しない**: デモなので上長不在のケースは対象外
- **代理承認者を設定**: 上長が代理を指名できる

[Answer]

考慮しない。

### 補足 H: 管理者と上長の兼任

[Question] 管理者ロールと上長を兼任することは可能ですか？（例: 人事部の部長が管理者でもある）

- **可能**: 管理者ロール + 上長フラグの両方を持てる
- **不可**: 管理者と上長は排他

[Answer]

はい。
